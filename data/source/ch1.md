## 第一章 初识智能体

欢迎来到智能体的世界！在人工智能浪潮席卷全球的今天，智能体（Agent）已成为驱动技术变革与应用创新的核心概念之一。无论你的志向是成为 AI 领域的研究者、工程师，还是希望深刻理解技术前沿的观察者，掌握智能体的本质，都将是你知识体系中不可或缺的一环。

因此，在本章，让我们回到原点，一起探讨几个问题：智能体是什么？它有哪些主要的类型？它又是如何与我们所处的世界进行交互的？通过这些讨论，希望能为你未来的学习和探索打下坚实的基础。

学完本章，你将能够：① 用一句话向朋友解释什么是 Agent；② 举出 3 个生活里的真实例子；③ 区分 5 种常见智能体类型，并说出 Workflow 与 Agent 的核心差别。

### 1.1 智能体 = 数字员工

你可以把智能体理解成一个能独立干活的“数字员工”。它不像普通程序那样只会按固定脚本走，而是会看情况、想步骤、动手做，最后还能根据结果调整自己。

我们用餐厅服务员来类比：服务员（智能体）的眼睛和耳朵是传感器，用来感知顾客点了什么、厨房出了什么菜；他的手和脚是执行器，用来上菜、倒水、结账；他的大脑负责判断“先给哪桌上菜”“这道菜是不是顾客点的”。整个过程就是一个“感知—思考—行动”的循环。

最小例子：自动恒温器就是一个最简单的智能体。它感知室温，如果温度低于设定值就启动加热，高于设定值就启动制冷。它虽然简单，但已经具备了“感知环境并自主行动”的雏形。

这和你有什么关系？未来你无论做产品、运营还是开发，都会遇到“让 AI 自动完成一件事”的需求。理解智能体，就是理解怎么把 AI 从“回答问题的聊天框”变成“能做事的助手”。

### 1.2 三个真实例子

为了消除“智能体听起来很玄”的感觉，我们看三个每天都会接触的真实例子。

例子一：智能音箱。你喊“播放周杰伦的歌”，它听到语音（感知），识别出你的意图（思考），然后调用音乐 App（行动），并把歌放出来。它不需要你每次都告诉它“先打开 App，再搜索歌手，再点击播放”。

例子二：扫地机器人。它通过激光雷达和摄像头感知房间布局，内部维护一张地图，规划清扫路径，遇到障碍物会绕开，电量低会自己回充。它会在清扫过程中不断更新地图，这就是一个基于模型的智能体。

例子三：手机上的 AI 助手。你让它“查一下明天北京天气，然后提醒我带伞”。它会调用天气工具拿到数据，再根据结果决定是否创建提醒。这已经是一个具备简单推理和工具使用能力的 LLM 智能体。

这三个例子共同说明：智能体离我们并不远，它正在以各种形态进入日常生活。

### 1.3 五分钟体验：天气查询智能体

最好的学习方式就是动手。接下来我们用 5 分钟做一个“天气查询智能体”。它会接收你的问题，调用天气 API 获取数据，再用自然语言回答你。

这个智能体虽然小，但已经包含了一个完整 Agent 所需的全部要素：感知（接收你的问题）、思考（决定调用天气工具）、行动（调用 API）、观察（拿到天气结果）。

你可以把它想象成一个专属小秘书：你问“今天上海天气怎么样？”，它不会凭空猜测，而是主动查资料，再整理成一句人话告诉你。这个过程体现了 Agent 最核心的能力——**遇到自己不知道的事，会主动使用工具去查**。

在下一页的代码中，我们把刚才的描述变成可以直接复制运行的 Python 代码。即使你没有编程基础，也可以先看懂它的逻辑：接收输入 → 选择工具 → 执行工具 → 返回结果。

### 1.3.1 可直接运行的天气查询智能体

下面是一个最小可运行的天气查询智能体示例。它使用免费的 wttr.in 接口，无需申请 API 密钥即可运行。

```python
import requests

def get_weather(city: str) -> str:
    url = f"https://wttr.in/{city}?format=j1"
    try:
        response = requests.get(url)
        response.raise_for_status()
        data = response.json()
        current = data['current_condition'][0]
        desc = current['weatherDesc'][0]['value']
        temp = current['temp_C']
        return f"{city}当前天气：{desc}，气温{temp}摄氏度。"
    except Exception as e:
        return f"查询失败：{e}"

class WeatherAgent:
    def __init__(self):
        self.tools = {"get_weather": get_weather}

    def run(self, question: str) -> str:
        # 简单理解：如果问题里出现城市名，就查天气
        for city in ["北京", "上海", "广州", "深圳", "杭州"]:
            if city in question:
                return self.tools["get_weather"](city)
        return "我还不会回答这个问题，试着问我某个城市的天气？"

agent = WeatherAgent()
print(agent.run("今天上海天气怎么样？"))
```

运行后你会看到类似“上海当前天气：晴，气温26摄氏度。”的输出。虽然只有几十行代码，但它已经展示了 Agent 的雏形：感知问题、选择工具、执行工具、返回结果。在后续章节中，我们会把这里的“简单理解”换成真正由大语言模型驱动的推理。

### 1.4 智能体 vs 程序

普通程序和智能体都能处理输入并给出输出，但它们的“做事方式”完全不同。普通程序像一条流水线：输入 → 固定处理 → 输出，每一步都是人写死的；智能体则像一个实习生：你告诉它目标，它自己观察、思考、行动，并根据反馈调整。

我们用三个维度对比：

- **输入**：程序需要严格格式化的输入，比如 JSON 或命令行参数；智能体可以理解自然语言，比如“帮我查一下明天上海会不会下雨”。
- **决策权**：程序只能执行预先写好的 if/else；智能体可以根据当前情况自己决定下一步做什么。
- **输出**：程序的输出通常是确定的数据；智能体的输出更自然、更灵活，还会解释为什么这样回答。

最小例子：一个“计算 two plus two”的程序只能处理英文和固定句式；而一个智能体可以听懂“2加2等于几”“二加二是多少”“帮我算一下2+2”等任意表达。

这就是关键区别：**程序执行规则，智能体追求目标**。

### 1.5 PEAS 框架

PEAS 是描述智能体任务环境的经典框架，四个字母分别代表 Performance（性能度量）、Environment（环境）、Actuators（执行器）、Sensors（传感器）。你可以把它理解为给智能体写一份“岗位说明书”。

以“智能健身教练”为例：

- **Performance**：用户的减脂/增肌目标达成率、训练完成率、用户满意度。
- **Environment**：用户的身体数据、可穿戴设备、家庭健身房、每日饮食记录。
- **Actuators**：语音指导、屏幕显示训练计划、发送提醒通知。
- **Sensors**：心率传感器、运动强度传感器、摄像头（动作捕捉）、用户输入的体重数据。

再以我们刚才的天气查询智能体为例：性能是回答是否准确；环境是天气 API 和用户的问题；执行器是返回文本答案；传感器是接收用户输入和 API 返回的 JSON 数据。

PEAS 的价值在于：它把“做一个智能体”这件模糊的事，拆成了四个可以具体讨论的部分。在 CH13 旅行助手项目中，我们会再次用它来拆解一个复杂应用。

### 1.6 智能体架构

[图片: 图 1.2 智能体架构示意图]

图 1.2 智能体架构示意图

一个完整的智能体通常由四层组成：感知层、推理层、行动层和记忆层。感知层负责接收外界信息；推理层负责分析、规划和决策；行动层负责调用工具或执行动作；记忆层负责保存历史对话、知识和经验。

你可以把智能体想象成一个人：眼睛耳朵是感知层，大脑是推理层，手脚是行动层，笔记本和长期记忆是记忆层。这四层缺一不可，共同支撑起一个能持续与环境交互的自主系统。

### 1.7 五种智能体类型总览

学术界通常把智能体按照内部决策架构分为五类，复杂度依次递增：简单反射型、基于模型的反射型、基于目标型、基于效用型、学习型。

这五种类型的关系就像学生的成长过程：

- 简单反射型：只会按固定规则反应，像一年级学生看到红灯就停。
- 基于模型反射型：会记住一些背景信息，像高年级学生能结合上下文理解老师的话。
- 基于目标型：会为了达成目标而规划，像中学生为了考试制定复习计划。
- 基于效用型：会权衡多个目标，像大学生在选课时要平衡兴趣、学分和就业。
- 学习型：会从经验中改进，像工作多年的专家越处理问题越顺手。

这五类并不是互相排斥的。现代 LLM 智能体往往同时具备多种特征，例如它既有目标，又能学习用户的偏好。

### 1.7.1 简单反射、基于模型与基于目标智能体

**简单反射智能体**只看当前输入，按固定规则立刻行动，不记过去也不想未来。生活类比：自动门感应到人就开，没感应就关；自动恒温器温度高就制冷，温度低就加热。它的优点是简单可靠、响应快；缺点是遇到规则没覆盖的情况就会失灵。

**基于模型的反射智能体**会维护一份对世界的内部记录，根据历史状态做决定。生活类比：开车时看后视镜和导航，而不只是看眼前的路。扫地机器人一边清扫一边更新房间地图，就是基于模型的典型例子。

**基于目标的智能体**以实现某个目标为导向，会主动选择能最快到达目标的行动。生活类比：导航软件，目标是目的地，会自动选最优路线。GPS 规划路径、棋类 AI 为了赢棋而布局，都是基于目标智能体。

这三类智能体的核心区别在于：简单反射“只看现在”，基于模型“记得过去”，基于目标“面向未来”。

### 1.7.2 基于效用与学习型智能体

**基于效用的智能体**不只追求目标，还要在众多方案里选最划算的一个。生活类比：精打细算的旅行者，会权衡时间、金钱和舒适度。例如，导航软件在推荐路线时，可能同时考虑距离、拥堵程度、过路费和风景，给每条路线打分，选择综合得分最高的。

**学习型智能体**能从经验里吸取教训，越做越好。生活类比：下棋越下越强的棋手，输多了就长记性。AlphaGo 通过与自己对弈数百万盘，不断改进策略，就是学习型智能体的代表。它的核心结构包括性能元件（做决定）和学习元件（根据反馈改进性能元件）。

这两类智能体更接近人类的决策方式：我们会权衡利弊，也会从失败中成长。现代 LLM 智能体通过 RLHF 等技术与人类偏好对齐，本质上也是一种学习。

### 1.8 LLM 出现以后

大语言模型（LLM）的出现，让智能体从“工程师 painstakingly 手写规则”变成了“用自然语言描述目标，模型自己推理”。传统智能体的能力边界取决于工程师写了多少规则；LLM 智能体的能力边界取决于模型本身的知识和推理能力。

对比时间线：

- 1950s-1980s：符号主义智能体，靠人工规则推理。
- 1990s-2010s：学习型智能体，靠强化学习从交互中学习。
- 2020s 至今：LLM 智能体，以大模型为大脑，配合工具、记忆和规划，完成复杂任务。

LLM 智能体带来的最大变化是“自然语言接口”。你不再需要把需求翻译成严格的代码逻辑，而是可以直接说“帮我规划一次厦门之旅”，模型会自动拆解任务、调用工具、动态修正。

这也意味着，未来设计智能体的核心技能，将从“写代码”逐渐转向“描述目标、设计工具、管理上下文”。

### 1.9 智能体工作原理

智能体的工作可以概括为一个循环：**感知 → 思考 → 工具 → 行动 → 记忆**。

1. **感知**：接收用户输入或环境反馈。例如用户问“北京今天天气怎样？”
2. **思考**：分析当前情况，决定下一步。模型会想：“用户问天气，我需要调用天气工具。”
3. **工具**：选择合适的工具并传入参数。例如调用 `get_weather(city="北京")`。
4. **行动**：执行工具，获取结果。例如 API 返回“晴，26°C”。
5. **记忆**：把这次交互保存下来，供下一轮使用。例如记住用户关心北京天气。

这个循环会不断重复，直到任务完成。在 ReAct 范式（CH4 会详细讲）中，这个循环被明确写成 Thought → Action → Observation → Thought → Action …… 每一步都清晰可见，便于调试。

### 1.10 核心组件

如果把智能体比作一个人，它的核心组件可以这样理解：

- **大脑**：大语言模型，负责理解、推理、决策。对应术语 LLM Agent 中的 LLM。
- **感官**：输入接口，包括用户消息、API 返回、传感器数据等。
- **手脚**：工具和执行器，例如天气 API、搜索引擎、代码解释器、机械臂。
- **记忆**：短期记忆（当前对话上下文）和长期记忆（向量数据库、知识库）。

最小例子：我们之前写的天气查询智能体中，Python 类的 `run` 方法相当于大脑做简单判断，`get_weather` 函数是手脚，`tools` 字典是工具箱。当后续接入真正的 LLM 后，“大脑”的推理能力会大幅提升。

这些组件在 CH7 自建框架、CH8 记忆检索、CH10 通信协议等章节会反复出现，是理解现代智能体架构的基础。

### 1.11 Workflow 与 Agent 的差异

Workflow 和 Agent 都能自动化任务，但它们的自由度不同。Workflow 是让 AI 按部就班地执行预先定义好的步骤；Agent 则是赋予 AI 自由度去自主达成目标。

生活类比：Workflow 像工厂流水线，每道工序已经排好顺序；Agent 像一位项目经理，你告诉它“月底前要上线新功能”，它自己拆解任务、协调资源、处理问题。

以电商退款为例：

- **Workflow**：金额 < 100 自动退；100-500 客服审核；>500 主管审批。每一步都是人写死的规则。
- **Agent**：系统理解退款政策，分析用户历史行为、商品状况、聊天记录，自主判断是否退款，并在不确定时询问人工。

什么时候用 Workflow？流程固定、规则清晰、容错要求高的场景，比如财务审批、合规检查。什么时候用 Agent？任务模糊、需要灵活推理、环境动态变化的场景，比如客服、研究、创意设计。

### 1.12 常见误区

刚开始接触智能体时，很容易产生一些误解。这里列出 5 个最常见的误区，帮你避开坑。

**误区一：Agent 什么都能做。** 实际上，Agent 的能力受限于它的工具、模型能力和上下文。没有工具，它只能聊天；工具不好，它也会犯错。

**误区二：Agent 完全不需要人管。** 现实中，Agent 仍然需要人设计目标、提供工具、监督边界。完全自主的 Agent 目前还很罕见。

**误区三：Workflow 已经过时了。** 并非如此。很多场景 Workflow 更可靠、成本更低。Agent 和 Workflow 不是替代关系，而是互补关系。

**误区四：Agent 不会出错。** Agent 也会幻觉、也会调用错工具、也会陷入死循环。因此需要设置最大循环次数、工具参数校验和人工审核。

**误区五：做 Agent 必须懂强化学习。** 对于 LLM Agent 来说，核心技能是提示工程、工具设计和上下文管理。强化学习是其中一条路径，但不是唯一路径。

### 1.15 智能客服实例

为了让你更直观地理解 Agent 循环，我们看一个智能客服的例子。

用户问：“我的订单怎么还没发货？”智能体的思考过程可能是：

- Thought：用户问订单发货情况，我需要先查订单状态。
- Action：调用 `query_order(user_id, order_id)`。
- Observation：订单已打包，预计明天发出。
- Thought：我知道了最新状态，现在需要安抚用户并给出明确时间。
- Action：Finish[“您的订单已在仓库打包完成，预计明天发出，请留意物流短信。”]

这个例子展示了 Thought-Action-Observation 循环在真实场景中的应用。它不是一次性回答，而是先查资料、再判断、再回复。这种“先核实再回答”的方式，正是 Agent 比单纯聊天机器人更可靠的原因。

### 1.16 工具调用是什么

工具调用（Tool Use）是智能体扩展能力的关键机制。大语言模型本身不会查天气、不会算数学题、不会访问你的数据库，但它可以通过调用外部工具来完成这些事。

生活类比：厨师不会种菜，但可以打电话叫外卖送食材；你不会修水管，但可以叫维修工上门。工具就是智能体的“外部专家”。

常见的工具类型包括：搜索引擎、天气 API、计算器、代码解释器、数据库查询、日历、邮件发送等。在 CH4 和 CH7 中，我们会学习如何给智能体设计工具、注册工具和解析工具的返回结果。

一个设计良好的工具应该具备三个特点：功能单一、输入输出明确、错误信息清晰。这样智能体才能稳定地调用它。

### 1.17 感知-思考-行动-观察循环

如果把智能体的工作流程画成图，就是一个不断旋转的循环：感知（Perception）→ 思考（Thought）→ 行动（Action）→ 观察（Observation）→ 再次感知。

- **感知**：接收外界信息，包括用户输入、工具返回、环境状态变化。
- **思考**：分析当前情况，决定下一步。对于 LLM Agent，这就是模型生成的一段内部推理。
- **行动**：基于思考，执行具体操作，通常是调用某个工具。
- **观察**：收集行动产生的结果，把它变成下一轮思考的输入。

最小例子：你问 Agent“上海和北京哪个更热？”。Agent 感知到问题后，思考需要分别查两个城市的温度；行动是两次调用天气 API；观察是两个温度值；最后思考比较大小并给出答案。

这个循环是智能体区别于传统程序的根本特征：它不是一次性的输入输出，而是持续与环境交互。

### 1.18 为什么智能体需要记忆

没有记忆的智能体就像一个“金鱼”，每次对话都从零开始。记忆让智能体能够：

- **保持上下文**：记住你之前说过的话，避免重复询问。
- **学习偏好**：记住你喜欢历史文化景点、预算范围、沟通风格。
- **积累经验**：记住哪些工具好用、哪些路径容易失败。

生活类比：短期记忆像脑子里的临时便签，聊完这轮就可能扔掉；长期记忆像硬盘里存的文件，关机后也不会丢。

在 CH8 记忆检索中，我们会详细学习短期记忆、长期记忆、向量数据库和 RAG。现在你只需要记住：记忆是智能体从“一次性工具”变成“长期伙伴”的关键。

### 1.19 多智能体协作

当任务足够复杂时，一个智能体可能忙不过来，这时可以让多个智能体协作。你可以把每个智能体想象成团队里的不同角色：一个负责调研，一个负责写作，一个负责检查。

生活类比：一个项目组里，策划、设计、开发各自分工又互相配合。多智能体系统也是如此，每个 Agent 有自己的职责，通过约定的协议交换信息。

在 CH10 通信协议中，我们会学习 MCP、A2A、ANP 等协议，它们就是 Agent 之间的“协作语言”。在 CH15 赛博小镇中，你会看到多个 Agent 如何在一个虚拟环境里互动。

多智能体协作的核心挑战是：如何分配任务、如何共享信息、如何处理冲突。这些问题目前仍是研究热点。

### 1.20 最小智能体代码骨架

下面是一个不含 LLM 的最小智能体代码骨架，帮助你理解 Agent 的核心结构。它包含感知、思考、行动和记忆四个方法。

```python
class SimpleAgent:
    def __init__(self):
        self.memory = []
        self.tools = {}

    def perceive(self, observation):
        self.memory.append(observation)

    def think(self):
        # 这里将来会接入 LLM 进行推理
        return "调用工具查询信息"

    def act(self, tool_name, **kwargs):
        if tool_name in self.tools:
            return self.tools[tool_name](**kwargs)
        return "未知工具"

    def run(self, query):
        self.perceive(query)
        thought = self.think()
        result = self.act("search", q=query)
        self.perceive(result)
        return result
```

这个骨架虽然目前还很简陋，但它展示了 Agent 的可扩展结构：你可以把 `think` 方法换成 LLM，把 `tools` 字典换成真实 API，把 `memory` 换成向量数据库。后续章节都是在这个骨架上不断丰富。

### 1.21 如何调试智能体

智能体比传统程序更难调试，因为它有循环、有随机性、有模型输出。以下是几个实用技巧：

1. **打印 Thought-Action-Observation 日志**：把每一步的输入输出都打印出来，定位问题所在。
2. **限制循环次数**：设置最大迭代次数，防止 Agent 陷入死循环或无限调用工具。
3. **固定随机种子**：在测试时固定 LLM 的温度和种子，让输出可复现。
4. **从小例子开始**：先用简单输入验证工具调用是否正确，再逐步增加复杂度。
5. **人工检查中间结果**：看看 Observation 是否被正确格式化，是否包含无关信息。

调试智能体的核心思路是：把黑箱拆成白箱。每一步的 Thought、Action、Observation 都应该是可见、可验证的。

### 1.22 从需求到智能体

当你拿到一个需求时，可以用下面四步把它转化为智能体设计：

1. **定义目标**：用户最终想要什么结果？例如“自动回复客服问题”。
2. **设计工具**：Agent 需要哪些外部能力？例如查询订单、查询物流、发送邮件。
3. **设计循环**：Agent 如何感知、思考、行动、观察？是否需要多轮交互？
4. **设定边界**：哪些情况必须转人工？最大循环次数是多少？错误如何处理？

以“智能会议纪要助手”为例：目标是生成会议纪要；工具包括语音识别、摘要模型、待办提取；循环是“录音 → 转文字 → 摘要 → 提取待办 → 输出”；边界是涉及敏感内容时提醒人工确认。

这个四步法会在 CH13 旅行助手项目中再次用到。

### 1.27 本章易错点

在开始学习下一章之前，请特别留意以下三个容易理解错的点：

1. **Agent 不是无所不能的**：它的能力取决于工具、模型和上下文。没有工具，它只能聊天；工具不可靠，它也会犯错。
2. **Workflow 和 Agent 不是非此即彼**：很多实际系统会混合使用，用 Workflow 处理固定流程，用 Agent 处理需要灵活推理的环节。
3. **五种智能体类型不是互斥的**：一个 LLM Agent 可以同时是目标驱动、效用权衡和学习改进的。分类是为了理解，不是为了贴标签。

这些易错点也会在后续章节的测验和项目中反复出现，建议你在笔记中重点标注。
