# 第十一章 Agentic-RL：从 SFT 到 GRPO 的 LLM 训练实战

## 本章定位

把大模型从"会聊天"升级到"会按目标办事"。当 prompt 调不动、提示词写到天花板时，就该让 Agent 接受训练：从示范里学（SFT），再从奖励里学（RLHF/GRPO）。本章用大白话讲清楚这套流程到底在做什么、为什么有效，以及怎么用 TRL 真跑一遍。

---

## 11.1 为什么 Agent 要"训练"：不只是 prompt

**大白话定义**：Agentic RL 是把"训练"这件事从静态的 prompt 工程，升级成可迭代的策略优化。Prompt 是给模型的"使用说明"，训练则是改模型本身的"工作习惯"。

**生活类比**：Prompt 像教新员工看员工手册，告诉他遇到这种情况要这么处理；训练则是让他跟着师傅干三个月，把判断力长到身体里。

**最小例子**：客服 Agent 用 prompt 写"客户生气时先道歉"，但模型还是会说错话；用 SFT 微调 1000 条优质工单后，模型不用 prompt 也能自然地先道歉再解决。

**和我有什么关系**：当你的 Agent 在生产环境里反复翻车、改 prompt 也没用时，就该考虑用训练替代调 prompt 了。这是 Agent 工程从"调参"走向"造脑"的分水岭。

---

## 11.2 训练数据怎么造

**大白话定义**：训练数据是 Agent 学习的教材。质量比数量更重要，格式比内容更关键，因为模型会原样模仿教材的样子。

**生活类比**：教小孩写作文，给他 100 篇范文比给他 1000 篇口水帖有用得多。关键不是字数，而是"格式整齐、答案标准"的范文。

**最小例子（GSM8K）**：用小学数学应用题 GSM8K 做教材：
- 训练集 7473 道题，每题带完整解题步骤（"Step 1: ... Step 2: ... Final Answer: 72"）
- 测试集 1319 道，模型从没看过
- 用准确率评估好坏，避免人工打分

**和我有什么关系**：选数据时优先选"答案可自动验证"的（如数学、代码、选择题），避免"答案模糊需要主观判断"的任务（如开放问答），否则训练信号噪声大、效果差。

---

## 11.3 数据格式转换

**大白话定义**：同一份数据要按训练方法切成不同格式：SFT 格式带"完整解题过程"，RL 格式只带"最终答案"。

**生活类比**：学徒模式（看师傅一步一步做）和学徒独立做（只告诉他"做出来什么样"），前者学格式，后者学思考。

**最小例子**：
- SFT 格式：`{"prompt": "48+24=?", "completion": "Step 1: 48+24=72. Final Answer: 72"}`
- RL 格式：`{"prompt": "48+24=?", "ground_truth": "72"}`（只给答案，不给过程，逼模型自己写推理）

**和我有什么关系**：SFT 是"打地基"，RL 是"盖楼"。没有 SFT 打下的格式基础，直接上 RL 训练会让模型连输出格式都学不会，奖励信号全是噪声。

---

## 11.4 SFT 监督微调：用示范教会 Agent

**大白话定义**：SFT 是给模型看"标准答案+完整思路"，让它照葫芦画瓢。这是所有后续训练的地基。

**生活类比**：新员工入职培训，老员工演示一遍"怎么接客户电话"，新人照着做几百遍，直到形成肌肉记忆。

**最小例子**：用 1000 条 GSM8K 完整解题过程训练 Qwen3-0.6B：
- 输入：问题 + 标准解法（含 Step 1、Step 2、Final Answer 标记）
- 输出：模型学会自动按这个格式答题
- 效果：原本乱说话的模型，输出变成结构化的"先分步再给答案"

**和我有什么关系**：SFT 投入产出比最高。花 1-2 小时用几百条示范数据微调，往往比写 100 行复杂 prompt 更有效。先把 SFT 跑出基线，再考虑 RL。

---

## 11.5 LoRA/QLoRA：低成本微调

**大白话定义**：LoRA 不动原模型参数，只训练两个小矩阵（低秩分解），参数量降到原来的 0.1%-1%，单卡就能跑。QLoRA 进一步把基座量化到 4-bit，显存再砍 75%。

**生活类比**：装修房子时，LoRA 像"不动承重墙只换软装"——便宜、快速、可逆；QLoRA 像"用更轻的家具"——把原本 8 吨的家具换成 2 吨的，省运费。

**最小例子**：
- 全量微调 7B 模型：需要 80GB 显存（4 张 A100）
- LoRA（r=8）：只需 16GB（1 张消费级 4090）
- QLoRA（4-bit 量化 + LoRA）：只需 6GB（MacBook 都能跑）

**和我有什么关系**：本地显卡跑不动全量微调时，QLoRA 是唯一可行的方案。记住三个常用值：rank=8（够用）、alpha=16（rank 的 2 倍）、target_modules=["q_proj","v_proj"]（注意力层）。

---

## 11.6 奖励模型：让 AI 知道什么算"好"

**大白话定义**：奖励模型是一个"打分器"，输入是模型的输出，输出是一个 0-1 的分数。它把"什么是好答案"这件事量化成数字，RL 才能据此优化策略。

**生活类比**：考试时老师批卷子——作文写得好不好，判卷老师说了算。奖励模型就是那个"老师"，只不过它是用神经网络模拟出来的。

**最小例子**：训练奖励模型时，给它看一对答案：
- 答案 A："48+24=70"（错误）
- 答案 B："48+24=72"（正确）
让模型学会"选 B"，并给出 B 的分数高于 A。

**和我有什么关系**：奖励函数设计是 RL 训练成败的关键。简单任务用准确率奖励（对错 0/1），复杂任务用组合奖励（准确率 + 长度 + 步骤分），避免"奖励黑客"（模型找漏洞刷分）。

---

## 11.7 RLHF 三步走

**大白话定义**：RLHF（Reinforcement Learning from Human Feedback）是用人类偏好训练模型的经典流程：先让人类给答案排序训练奖励模型，再用奖励模型指导强化学习。

**生活类比**：餐厅培训新厨师：① 顾客对菜品打分（收集偏好）；② 老板根据打分总结出"什么叫好吃"（训奖励模型）；③ 新厨师根据"好吃标准"改进厨艺（RL 优化）。

**最小例子（三步）**：
1. **收集偏好数据**：让人类对同一问题的多个答案排序（A>B>C）
2. **训练奖励模型**：用排序数据训练一个打分网络
3. **PPO 强化学习**：用奖励模型的分数作为信号，用 PPO 算法调整 LLM

**和我有什么关系**：RLHF 是 ChatGPT 出圈的核心技术，但它流程复杂、需要人类标注。如果你的任务有"客观标准"（如数学、代码），可以直接用规则奖励跳过第一步，更便宜。

---

## 11.8 PPO 算法：用奖励信号调整 Agent

**大白话定义**：PPO（Proximal Policy Optimization）是最经典的 RL 算法，它通过限制"每次更新幅度"来稳定训练，避免模型一次跑偏就再也回不来。

**生活类比**：老司机学车——教练每次只让"微调"一点点方向盘，而不是让你突然 90 度转弯，否则会翻车。PPO 就是"每次只微调一点点"。

**最小例子**：PPO 更新策略的核心公式：
- 计算新策略和旧策略的"比率" ratio
- 用 clip(ratio, 0.8, 1.2) 限制变化幅度
- 超过这个范围的更新被截断，不让模型"动作变形"

**和我有什么关系**：PPO 稳定但慢、贵（要训 4 个模型：策略、参考、价值、奖励）。如果是 LLM 训练，优先用 GRPO 替代，省一半显存且更稳。

---

## 11.9 GRPO：更稳的策略优化

**大白话定义**：GRPO（Group Relative Policy Optimization）是 DeepSeek 提出的 PPO 简化版。它去掉了"价值模型"，改用"组内相对奖励"代替绝对奖励。

**生活类比**：班级考试不公布绝对分（避免大家比较），只告诉你"你比班级平均分高多少"——这样既能激励学生，又避免了"卷王永远赢"的极端情况。

**最小例子**：对同一道题生成 4 个答案：
- 4 个全对 → 组内奖励都是 0（标准，无需强化）
- 3 对 1 错 → 对的得 +0.25，错的得 -0.75
- 4 个全错 → 全部惩罚，不强化

**和我有什么关系**：GRPO 是当前 LLM RL 训练的事实标准（DeepSeek-R1 用它训练）。显存省一半，训练更稳定，是 LLM 训练的首选 RL 算法。

---

## 11.10 KL 散度惩罚

**大白话定义**：KL 散度是衡量"两个模型有多像"的指标。KL 惩罚越大，模型越不能离 SFT 模型太远，避免"训练 RL 反而把基础能力搞丢了"。

**生活类比**：学钢琴时不能为了练新曲子就丢掉基础指法。KL 惩罚就是"指法保留条款"——鼓励你学新东西，但别忘了基本功。

**最小例子**：
- kl_coef=0.01：模型可以大幅改动，但可能丢了 SFT 学到的格式
- kl_coef=0.1：平衡探索和稳定（推荐）
- kl_coef=0.5：模型几乎不动，训练等于白做

**和我有什么关系**：监控训练时，KL 散度 > 0.5 是危险信号——说明模型"飘了"。需要降低学习率或增大 kl_coef。

---

## 11.11 跟着做：用 TRL 微调一个 ReAct Agent

**大白话定义**：TRL（HuggingFace Transformer Reinforcement Learning）是工业级 RL 训练库，封装了 SFT/PPO/GRPO/DPO 一整套实现，开箱即用。

**生活类比**：TRL 像"自动挡汽车"——你只要告诉它往哪走（目标），换挡踩油门（参数）它自己处理，不用懂发动机原理。

**最小例子（5 行代码微调）**：
```python
from trl import SFTTrainer, GRPOTrainer
from datasets import load_dataset

dataset = load_dataset("gsm8k", split="train[:1000]")
trainer = SFTTrainer(model="Qwen3-0.6B", train_dataset=dataset)
trainer.train()  # 跑 SFT
```
- 装：`pip install trl peft accelerate`
- 跑 SFT：几行代码搞定
- 跑 GRPO：再几行代码

**和我有什么关系**：不要自己手写 RL 训练循环。TRL 帮你处理分布式、混合精度、梯度累积等所有工程细节，让你专注业务逻辑。

---

## 11.12 模型评估与分析

**大白话定义**：训练完不等于能用——必须用独立测试集评估模型真实能力。评估分三层：准确率（对不对）、效率（多快多省）、质量（推理是否清晰）。

**生活类比**：考试不光看分数（准确率），还要看用了多少时间（效率）、解题过程是否规范（质量）。三个维度都好才是真优秀。

**最小例子**：训练完 Qwen3-0.6B 后评估：
- 准确率：GSM8K 测试集上答对的比例（典型 40-60%）
- 平均长度：平均生成多少 token（影响成本）
- 格式正确率：是否按"Step 1... Final Answer..."输出

**和我有什么关系**：评估比训练更重要。一个"训练时漂亮但测试时翻车"的模型没价值。永远保留 20% 数据做测试集，不参与训练。

---

## 11.13 错误分析

**大白话定义**：错误分析是把模型答错的题分类，看它在哪些类型上反复出错，从而对症下药。

**生活类比**：医生看病——光知道"病人发烧"不够，要进一步查是细菌感染还是病毒感染，才能开对药。

**最小例子（4 类错误）**：
- **计算错误**（42%）：推理对但算错 → 用计算器工具
- **推理错误**（24%）：思路本身错 → 换训练数据
- **理解错误**（29%）：题没读懂 → 强化 prompt
- **格式错误**（5%）：答案对但格式乱 → 补 SFT

**和我有什么关系**：错误分析比"总准确率"更有指导意义。看到 42% 计算错误，优先考虑外挂计算器工具，而不是再训 1000 条数据。

---

## 11.14 超参数调优

**大白话定义**：超参数是训练前手动设定的"开关"（学习率、LoRA 秩、KL 系数等）。调参不是玄学，是有方法论的实验科学。

**生活类比**：学骑自行车时调座椅高度——太低够不着踏板，太高摔跤。超参数调优就是找"刚好合适"的位置。

**最小例子（3 种方法）**：
- **网格搜索**：穷举所有组合，稳但慢
- **随机搜索**：随机抽 N 组，快但可能漏掉最优
- **贝叶斯优化**：用历史结果指导下一组，最智能（用 Optuna 库）

**和我有什么关系**：不要一次调所有参数。先用默认值跑通流程，再逐个调一个最不确定的。学习率和 KL 系数是 LLM 训练最敏感的两个，先调它们。

---

## 11.15 分布式训练与部署

**大白话定义**：模型变大、数据变多时，单卡跑不动，需要把训练任务分到多张卡上（分布式训练），跑完后再把模型部署成服务（推理服务化）。

**生活类比**：搬家时自己搬太慢，请搬家公司（多工人协作）；搬完布置好房间（部署好服务）。

**最小例子**：
- 2-8 卡：用 DDP（数据并行），配置简单
- 大模型（>7B）：用 DeepSpeed ZeRO-3，省显存
- 部署：合并 LoRA 权重 + 8-bit 量化 + FastAPI 包装

**和我有什么关系**：本地只有 1 张卡？优先用 QLoRA 跑 7B-13B 模型。多卡训练不一定要，DDP 配 4 行 yaml 就能用。

---

## 11.16 训练常见问题与排错

**大白话定义**：训练 RLHF/GRPO 时常见三大问题——奖励不涨、KL 爆炸、生成质量下降。每种问题都有对应的诊断和修复方法。

**生活类比**：开车时水温过高、引擎异响、刹车失灵——每个症状对应不同的故障点和修法。

**最小例子**：
- **奖励不涨**：学习率太小或 KL 系数太大 → 增大 lr 到 5e-5，减小 kl_coef 到 0.05
- **KL 爆炸（>0.5）**：学习率太大 → 降低 lr 到 1e-5，或增大 kl_coef 到 0.1
- **生成质量下降**：奖励函数过拟合 → 换组合奖励或加 KL 惩罚

**和我有什么关系**：训练出问题不要慌，按这 3 个症状排查 80% 的情况。监控时盯紧"奖励曲线 + KL 散度 + 生成样本"三件套。

---

## 11.17 本章小结

**大白话定义**：Agentic RL 不是魔法，是一套标准化的"打地基-盖楼-装修"流程：SFT 打地基、RL 盖楼、奖励函数设计装修。

**生活类比**：装修房子不可能跳过水电（SFT）直接搞软装（RL），否则住进去就漏水。本章讲的训练流程就是这个"装修顺序"。

**最小例子（5 步走）**：
1. 准备数据：选"答案可验证"的任务（数学、代码、选择）
2. SFT 微调：100-1000 条示范数据，3 个 epoch
3. 设计奖励：准确率 + 长度惩罚 + 步骤奖励
4. GRPO 训练：从 SFT 模型继续，学习率 1e-5
5. 评估部署：测试集验证 + LoRA 合并 + 量化部署

**和我有什么关系**：未来 2 年，Agent 工程师的核心竞争力是"会用 RL 训练 Agent"。这一章的内容是从 prompt 工程师升级到 Agent 训练师的入场券。
