## 第三章 大语言模型基础

想象你身边坐着一个读过几千万本书、但没有自己记忆的超级读者——你随口问一句，他就能顺口答出一段。这就是**大语言模型（Large Language Model，简称 LLM）**的日常。

前两章我们分别介绍了智能体的定义和发展历史，本章将聚焦于大语言模型本身。我们不堆公式，只回答三个问题：LLM 是什么？它怎么来的？它怎么用？通过这一章，你会为后续真正动手搭建智能体打下最关键的“大脑”基础。

学完本章，你将能够：① 用一句话说清楚 LLM 是什么；② 解释 Transformer 三件套各自的作用；③ 区分预训练、SFT、RLHF、蒸馏四个训练阶段；④ 设计一个能跑出好结果的提示词；⑤ 知道主流模型之间的差别，以及如何选型。

## 3.1 本章概述

LLM 不是一个“黑箱魔法”，它本质是一个“猜下一个字”的程序。本章我们会沿着这条线索展开：

- **它是什么**：会读会写的文字预测器，核心技能是“猜下一个字”。
- **它怎么来的**：从统计模型一路进化到 Transformer，再被“喂”到几 TB 文本里训练出来。
- **它怎么用**：怎么向它提问、怎么选模型、怎么避开它一本正经地胡说八道。

[提示] 后面会带你亲自跑一个小模型，5 分钟就能体验“调教”它。

## 3.2 LLM 是什么：会读会写的文字预测器

**大白话定义**：大语言模型就是一个“猜下一个字”的程序。你给它前半句，它把后半句补上；你问它问题，它就把答案“补”出来。

**生活类比**：把它想成一个超级勤奋的“接龙高手”。小时候我们玩成语接龙——听到“红”，脑子里会自动蹦出“苹果”。LLM 就是把这种“自动接词”做到了极致，只不过它会接的不只是成语，而是整段整段的话。

**最小例子**：
输入：“今天天气很好，我们去”
模型会继续写：公园散步 / 爬山 / 逛街……
它做的事情，本质上就是“一字接一字地续写”。

**和我有什么关系**：你让 ChatGPT 帮你写邮件、翻译、答疑，背后都是它在不停做“猜下一个字”这件事。明白这个本质，你就能理解为什么它有时一本正经胡说八道——它不是在“查资料”，而是在“续写最可能的句子”。

## 3.3 为什么叫“大”：参数量意味着什么

**大白话定义**：参数（Parameter）就是模型里那些可以调节的“小旋钮”，模型在训练中不断拧这些旋钮，让输出越来越像人话。所谓“大”，就是旋钮多到几亿、几千亿个。

**生活类比**：把模型想成一个灯泡厂。旋钮越多、灯泡越亮，能表达的语言细节就越丰富。10 万旋钮的模型像一只小夜灯，只能说“早安”；700 亿旋钮的模型像太阳，能写诗、写代码、做推理。

**最小例子**（常见量级参考）：
- 0.5B（5 亿）：能在手机/笔记本上跑，回答日常问题
- 7B–13B（70–130 亿）：常见开源模型，能写代码、做总结
- 70B+：准专业级，复杂任务表现稳定
- 数百 B–千 B（GPT-4 / Claude 4 / Gemini 2.5）：闭源旗舰级

**和我有什么关系**：选模型时，参数规模直接决定你要付出的成本和能获得的能力上限。要本地跑，选小一点；要性能顶满，就用 API 调大模型。

## 3.4 训练四步走：预训练 → SFT → RLHF → 蒸馏

**大白话定义**：一个能用的 LLM 一般要经历四个阶段，先变成“读书破万卷”的文盲，再被教成会听话的助手。

**生活类比**：把训练 LLM 想成培养一个翻译：
1. **预训练**（广泛阅读）：让它读几 TB 网页、书、新闻，只是“续写下一个字”，把世界知识装进脑子。
2. **SFT 监督微调**（拜师学艺）：拿几千几万条“问题–好答案”对，让它模仿“该这么答”。
3. **RLHF 人类反馈强化学习**（师傅打分）：人对多个答案排序，模型学着挑出更讨人喜欢的回答，更安全、更有用。
4. **蒸馏**（师傅带徒弟）：让小模型去模仿大模型的输出，得到一个又快又便宜的小号版本。

**最小例子**：你手机里的输入法联想、Google 的搜索补全，背后都跑着蒸馏过的小模型。

**和我有什么关系**：知道这四步，你就能在选模型时看懂商家宣传——所谓“对齐”就是 SFT+RLHF，所谓“轻量版”就是蒸馏。

## 3.5 Transformer 三件套：注意力 + 前馈 + 残差

**大白话定义**：现代 LLM 几乎都长一个样——把“三件套”一层一层堆起来，这就是 Transformer。它的核心三件套是：**自注意力、前馈网络、残差+层归一化**。

**生活类比**：把 Transformer 想象成一家餐厅后厨：
- **自注意力**：顾客点了“酸菜鱼”，大厨同时看向所有订单，决定这道菜最该参考哪几道。
- **前馈网络**：大厨拿到信息后开始切菜、调味——把“别人的经验”变成“自己的手艺”。
- **残差+层归一化**：每做完一步，保留一下原味（残差），同时让火候稳定（归一化），这样做了几十道菜还不串味。

**最小例子**：一句话“我喜欢吃苹果”，注意力发现“吃”和“苹果”关系最强；前馈网络据此把“吃”翻译成动作含义；残差和归一化让“喜欢”这种情感信息不丢失。

**和我有什么关系**：所有主流 LLM（GPT、Claude、Gemini、Qwen）都是这个结构。搞懂它，你就抓住了 90% 现代 AI 的骨架。

## 3.6 自注意力：每个字都能“看见”全文

**大白话定义**：自注意力（Self-Attention）就是让句子里的每个字，都“瞟一眼”所有其他字，看看谁和自己最相关。

**生活类比**：就像开会时，每位同事听别人发言，都会默默在心里给其他人打分——谁说的和自己工作最相关，就多听几句。Transformer 给每个字都做了这件事。

**最小例子**：句子 “The animal didn't cross the street because **it** was tired.” 当模型读到“it”的时候，自注意力会让“it”多看看“animal”，于是模型就明白“it”指代 animal 而不是 street。

**工作过程（用人话讲）**：
1. 每个字都生成三个角色：Q（要问什么）、K（我是谁）、V（我有什么内容）。
2. 用 Q 去跟所有 K 比一比，得到“谁跟我最相关”的分数。
3. 把分数变成权重，再去按比例“取走”V。

**和我有什么关系**：这就是机器能理解代词指代、上下文关联的关键。你看到的对话能力、阅读理解能力，全靠它。

## 3.7 多头注意力：多位专家协同判断

**大白话定义**：单头注意力一次只看一种关系，而语言里的关系五花八门——指代、时态、从属、并列……多头注意力就是请来多位“专家”，每人看一种关系，最后汇总。

**生活类比**：医院会诊。一位医生可能只看“心脏”，但几位医生一起看，才能兼顾心、肺、脑。同理，多头注意力把输入切成多份，并行算多个“关系视图”。

**最小例子**：句子“她买了一个红色的苹果，因为**它**很甜”。
- 头 A 关注：指代关系——“它”→ 苹果。
- 头 B 关注：修饰关系——“红色的”→ 苹果。
- 头 C 关注：因果关系——“因为很甜”→ 买了它。

三种角度拼起来，模型才真正“读懂”了这句话。

**和我有什么关系**：头数（num_heads）越大，模型越能抓细节，但代价是算力翻倍。这也是为什么小模型“头少”，大模型“头多”。

### 多头注意力 PyTorch 实现

```python
import torch
import torch.nn as nn
import math

class MultiHeadAttention(nn.Module):
    def __init__(self, d_model, num_heads):
        super().__init__()
        assert d_model % num_heads == 0
        self.d_model = d_model
        self.num_heads = num_heads
        self.d_k = d_model // num_heads
        self.W_q = nn.Linear(d_model, d_model)
        self.W_k = nn.Linear(d_model, d_model)
        self.W_v = nn.Linear(d_model, d_model)
        self.W_o = nn.Linear(d_model, d_model)

    def scaled_dot_product_attention(self, Q, K, V, mask=None):
        # Q, K, V shape: (batch, heads, seq, d_k)
        scores = torch.matmul(Q, K.transpose(-2, -1))
        scores = scores / math.sqrt(self.d_k)
        if mask is not None:
            scores = scores.masked_fill(mask == 0, -1e9)
        attn_weights = torch.softmax(scores, dim=-1)
        return torch.matmul(attn_weights, V)

    def split_heads(self, x):
        b, s, d = x.size()
        return (x.view(b, s, self.num_heads, self.d_k)
                .transpose(1, 2))

    def forward(self, Q, K, V, mask=None):
        Q = self.split_heads(self.W_q(Q))
        K = self.split_heads(self.W_k(K))
        V = self.split_heads(self.W_v(V))
        attn = self.scaled_dot_product_attention(Q, K, V, mask)
        # 合并多头: (b, h, s, d_k) -> (b, s, d_model)
        b, h, s, dk = attn.size()
        attn = attn.transpose(1, 2).contiguous()
        attn = attn.view(b, s, self.d_model)
        return self.W_o(attn)
```

## 3.8 位置编码：让模型知道字的顺序

**大白话定义**：自注意力有个“毛病”——它把句子当成“一堆字”而不是“一串字”，顺序颠倒也照样算。位置编码就是给每个字贴个“座位号”，让模型知道谁先谁后。

**生活类比**：想象 6 个人同时大声说话，自注意力让你听见每个人在说什么，但分不清谁先开口。位置编码就是给每人发个编号牌，模型才听得懂先后顺序。

**最小例子**：
- 没位置编码：“狗咬人” ≡ “人咬狗”（模型分不清）
- 有位置编码：模型能正确理解主语、宾语。

**常见做法**：原版 Transformer 用 sin/cos 函数生成一组固定的位置向量；现在很多模型也用“可学习的位置编码”，效果差不多。

**和我有什么关系**：上下文窗口越长，位置编码设计就越关键——这正是为什么超长上下文模型（如 100 万 token）需要专门优化位置信息。

## 3.9 残差连接与层归一化：让深层网络也能训练

**大白话定义**：当 Transformer 堆到几十层、上百层时，光靠注意力是不够的——会出现“信号传着传着就没了”的情况。残差连接 + 层归一化就是两剂“稳定剂”，让深层网络还能稳训练。

**生活类比**：把上百层的网络比作一条 100 人的传话队伍。
- **残差连接**：每传一段，原话也抄一份贴上去——下游的人既能听到上游加工过的版本，也能看到“最初的原文”，信息不丢。
- **层归一化**：每过几层，就让数据“归一化”——把大家说话的音量统一一下，不然后面的人要么听不见、要么被震聋。

**最小例子**（公式化讲）：
- 残差：output = x + Sublayer(x)
- 层归一化：把每一行的均值变成 0、方差变成 1

**和我有什么关系**：正是因为这套设计，LLM 才能堆到几百层、参数堆到几千亿仍能稳定训练。深度学习能“深”，这两招功不可没。

## 3.10 Decoder-Only：只做“下一字预测”的简洁架构

**大白话定义**：现代大语言模型（GPT、Claude、Qwen）几乎都是 Decoder-Only 架构——它只关心一件事：给定前面的字，预测下一个字。

**生活类比**：把 Decoder-Only 模型想成一个“接龙狂魔”，永远只问“我已经写的这些字后面，最可能接什么”。它没有“理解整个句子再回答”的部分，它就是一路续写。

**最小例子**：
输入：'Datawhale Agent is'
→ 模型续 'a' → 'a powerful' → 'a powerful tool' ……
每写一个字，就把它追加回输入，再写下一个。

**关键机制：掩码自注意力**——写第 N 个字时，模型“数学上”被禁止偷看第 N+1 之后的字。这就保证它真的是“一字一字往前推”。

**和我有什么关系**：这种“看似简陋”的架构，反而成了大模型时代的标配——简单、统一、易扩展、还天然适合对话和生成任务。

## 3.11 提示工程：怎么和 LLM 说话

**大白话定义**：提示工程（Prompt Engineering）就是“和 LLM 说话的技巧”。同样的问题，换一种问法，答案能差出十万八千里。

**生活类比**：把 LLM 想象成一个能力很强但有点木讷的实习生——你问得越具体、给得越多上下文，他干得越好。问得太抽象，他只能瞎猜。

**最小例子**：
- 差问法：“写个广告。”（模型只能随便写）
- 好问法：“请以‘夏日冰饮’为主题，写一句 15 字以内的朋友圈文案，语气活泼。”（模型立刻有方向）

**四个常用技巧**：
1. **角色扮演**：开头告诉它“你是有 10 年经验的营养师”，它会按这个身份回答。
2. **给示例**：把“问题-答案”示例摆上两三组，模型立刻学会格式。
3. **分步指令**：让模型“一步步思考”，复杂任务效果明显变好。
4. **加约束**：明确输出长度、格式、风格，模型更听话。

**和我有什么关系**：写好提示词，几乎是普通人用好 LLM 的唯一“硬技能”。一个普通模型+好提示，常胜过顶级模型+烂提示。

## 3.12 采样参数：温度、Top-k、Top-p

**大白话定义**：模型每次都会给所有可能的“下一个字”打分（概率），采样参数就是决定“按什么规则从这些候选里挑一个”。

**生活类比**：把模型想成一个写稿人，眼前有一沓候选字，采样参数决定他从这沓里“严格挑”还是“随便抽”。

**三个最常用参数**：
- **温度 Temperature**：调“胆子”。温度低 → 模型变得保守、专挑最可能的字（适合写代码、写事实）；温度高 → 模型变放飞、敢选冷门字（适合写诗、头脑风暴）。
- **Top-k**：只保留概率最高的 k 个字，从这 k 个里抽。k=1 就是“永远挑最可能那个”，输出会很死板。
- **Top-p（核采样）**：保留累计概率刚到 p 的那批字，p=0.9 就是“从占 90% 概率的字里挑”。它比 Top-k 更灵活，能根据分布自动伸缩。

**最小例子**：写代码时温度设 0.1（更稳）；写营销文案时温度 0.8（更有创意）。

**和我有关系**：几乎所有 LLM 工具都把这三个参数暴露在界面上。改一改，立刻能看到输出风格的变化。

## 3.13 零样本、少样本与思维链

**大白话定义**：根据“给不给示例、给多少”，提示可以分成三类。

**生活类比**：让实习生写报告。
- **零样本（Zero-shot）**：直接说“写一份季度总结”，不提供任何范文。
- **少样本（Few-shot）**：先给他两篇范文，再说“按这个调调写”。
- **思维链（Chain-of-Thought）**：复杂题直接给答案容易错，那就补一句“请你一步步思考”，他会写出推理过程再给结论，准确率常常翻倍。

**最小例子**（数学题）：
- 零样本：“篮球队一赛季 80 场赢 60%，下赛季 15 场赢 12 场，总胜率？” → 模型可能瞎猜。
- 思维链：再加一句“**请一步步思考**”，模型会写出 80×0.6=48、95 场赢 60 场、60/95≈63.16%，又准又清晰。

**和我有关系**：
- 简单任务用零样本，省事；
- 复杂任务用思维链，效果提升明显。
这是用好 LLM 的“性价比最高”两招。

## 3.14 上下文窗口：LLM 的工作记忆

**大白话定义**：上下文窗口（Context Window）就是模型一次能“装下”多少文字的容量，超过这个容量，前面说的话它就忘了。

**生活类比**：人的工作记忆也有限——让你背一串 20 个数字你能复述，让你背 1 万个你肯定记不住。LLM 也是一样，它的“工作记忆”用 token 衡量。

**常见量级**：
- 老模型：2K–8K token（约 2000 中文字）
- 主流：32K–128K token
- 长文本旗舰：1M token（够塞一整本小说）

**最小例子**：你让模型读 50 页合同然后问问题。
- 合同 < 上下文窗口：模型能引用细节。
- 合同 > 上下文窗口：模型只能“凭印象”，容易答错。

**和我有关系**：
- 处理长文档、整本电子书、代码库 → 选大窗口模型。
- 写长对话/Agent 时注意“会话别太长”，超出窗口的早期内容会被自动丢弃。
理解这一点，你就知道为什么有的对话“聊久了 AI 就变笨了”。

## 3.15 主流大模型一览：GPT、Claude、Gemini、文心、通义

**大白话定义**：目前国内外能打的“大模型”可以数出十几个，但真正有影响力的，主要有五家。

**生活类比**：把市场想成“手机品牌”——各家有各家的旗舰、各有各的粉丝。

**五大代表**：
- **GPT（OpenAI，美国）**：ChatGPT 背后的模型。最通用，生态最丰富，闭源。
- **Claude（Anthropic，美国）**：长文档、写作、代码表现稳定，被很多企业用户偏爱。
- **Gemini（Google，美国）**：原生多模态见长，能看图、看视频、看屏幕。
- **文心一言（百度，中国）**：中文场景打磨深，搜索、政务、本地化能力强。
- **通义千问（阿里，中国）**：开源 Qwen 系列表现亮眼，中文开源生态最活跃。

**最小例子**：你做英文翻译、写正式邮件——GPT/Claude；你做中文公文、本地化产品——文心/通义；你做视频/图像理解——Gemini 表现更稳。

**和我有关系**：选模型不是看“谁最强”，而是看“谁最合适你的任务+语言+预算”。下一节我们还会用一张表系统对比。

### 主流大语言模型对比

| 代表 | GPT-5, Claude 4, Gemini 2.5 | Llama 4, Qwen, Mistral |
| 性能 | 领先，最前沿能力 | 接近闭源，持续追赶 |
| 成本 | API 按 Token 计费 | 自备硬件，无 API 费用 |
| 部署 | 开箱即用，无需运维 | 本地部署，数据不离开 |
| 可控性 | 受服务商条款限制 | 完全自主控制 |
| 定制化 | 有限（Prompt/微调） | 可任意微调修改 |
| 上下文 | 较大（128K~1M Token） | 差异大（8K~1M Token） |
| 生态 | 成熟稳定，文档完善 | 社区活跃，工具丰富 |

## 3.16 多模态：让 LLM 看图听声音

**大白话定义**：传统 LLM 只会处理文字，多模态（Multimodal）模型则是“又看又听又说”——能同时理解图片、音频、视频。

**生活类比**：从前的 AI 像“只读文字书的人”，多模态 AI 像“会看漫画、会听歌、还会看小视频的人”，感知维度一下子丰富起来。

**最小例子**：
- 拍一张超市小票照片，丢给多模态模型问“这一单总共多少？”
- 把一段英语会议录音丢进去，让它生成中文纪要。
- 把 UI 截图发过去，让它“按这个界面写前端代码”。

**常见模态能力**：
- 视觉：图像理解、OCR、图表问答、UI 截图转代码
- 语音：语音转写、口语翻译、情感识别
- 视频：长视频理解、关键片段提取

**和我有关系**：做智能体时，多模态往往比“纯文本”更实用——很多真实任务本身就带着图、声音、视频。
把“文字能力”升级成“多模态能力”，是当前 Agent 产品最值得投入的方向之一。

## 3.17 文本分词：把文字变成数字

**大白话定义**：模型本质是个数学函数，只认数字不认识字。分词（Tokenization）就是把一句话切成“模型认识的最小单位”，再给每个单位编个号。

**生活类比**：把分词想成“把中文句子拆成语素表，再翻成数字密码”。模型看到的不是“你好”，而是 [12345, 67890] 这样的数字串。

**三种常见做法**：
- **按词分**：每个词一个 token，但词表会爆炸、生词不认识。
- **按字分**：每个字一个 token，词表小，但语义弱、训练慢。
- **子词分词（Subword）**——现代主流：常见词整个留下，生词切成片段。
例：'Tokenization' → ['Token', 'ization']
常见算法有 BPE（GPT 系列）、WordPiece（BERT）、SentencePiece（Llama）。

**最小例子**：
- 英文 'unhappiness' 切成 ['un', 'happiness']
- 中文 “大语言模型” 可能切成 ['大', '语言', '模型']

**和我有关系**：API 按 token 计费、中英文 token 数差异巨大——懂分词，你才能算清账单、控住成本。

### BPE（字节对编码）算法

```python
import re, collections

def get_stats(vocab):
    """统计相邻词元对频率"""
    pairs = collections.defaultdict(int)
    for word, freq in vocab.items():
        symbols = word.split()
        for i in range(len(symbols) - 1):
            pairs[symbols[i], symbols[i+1]] += freq
    return pairs

def merge_vocab(pair, v_in):
    """合并最高频词元对"""
    v_out = {}
    bigram = re.escape(' '.join(pair))
    p = re.compile(r'(?<!\S)' + bigram + r'(?!\S)')
    for word in v_in:
        w_out = p.sub(''.join(pair), word)
        v_out[w_out] = v_in[word]
    return v_out

# 迷你语料库（</w>表示词尾）
vocab = {'h u g </w>': 1, 'p u g </w>': 1,
         'p u n </w>': 1, 'b u n </w>': 1}

for i in range(4):
    pairs = get_stats(vocab)
    if not pairs:
        break
    best = max(pairs, key=pairs.get)
    vocab = merge_vocab(best, vocab)
    print(f"第{i+1}次合并: {best} -> {''.join(best)}")
# 输出:
# 第1次合并: ('u', 'g') -> ug
# 第2次合并: ('ug', '</w>') -> ug</w>
# 第3次合并: ('u', 'n') -> un
# 第4次合并: ('un', '</w>') -> un</w>
```

## 3.18 分词器对开发者的意义

**大白话定义**：分词器（Tokenizer）就是负责把文字变数字的小工具，看起来不起眼，但直接影响模型的“智商”和你的钱包。

**三个关键影响**：
1. **上下文窗口按 token 算**：同样一段话，中文可能比英文“贵”几倍。同样一篇文章，中文消耗的 token 数会更多。
2. **API 计费按 token**：发送 1000 token 和发送 1000 汉字，账单可能差几倍。
3. **模型表现的隐藏陷阱**：
   - '2 + 2' 和 '2+2' 在某些分词器下会被切成不同的 token，模型表现天差地别。
   - 'Apple' 和 'apple'（首字母大小写）可能被切成完全不同的 token。
   - 长难词、emoji、罕见字符，最容易踩坑。

**和我有关系**：
- 做 Agent 时务必在提示词里保持一致的格式（多余空格、奇怪换行尽量去掉）。
- 写代码时显式调用分词器，先看看到底切成什么样。
- 长对话注意 token 累计——聊久了账单会变贵，模型也会变“笨”。

## 3.19 调用开源大语言模型（HuggingFace）

**大白话定义**：除了调用 GPT/Claude 这种“付费 API”，你还可以把模型直接下载到本地自己跑，这就是开源 LLM。HuggingFace 是目前最方便的开源模型“超市”。

**生活类比**：调用闭源 API 就像“点外卖”——好吃、方便，但菜从别人厨房出。开源 LLM 就像“自己买菜回家做”——麻烦点，但食材放心、调料自己掌握。

**最小调用流程**（HuggingFace Transformers）：
1. 安装：`pip install transformers torch`
2. 加载分词器：`tokenizer = AutoTokenizer.from_pretrained("模型名")`
3. 加载模型：`model = AutoModelForCausalLM.from_pretrained("模型名")`
4. 把文字编码成 token ID。
5. `model.generate(...)` 生成回答。
6. 把 token ID 解码回文字。

**最小例子**（5 亿参数 Qwen）：模型能在笔记本电脑上跑，问它“你好，请介绍自己”，它会流畅回答。

**和我有关系**：处理敏感数据、想离线使用、想自己微调——本地部署开源模型是必由之路。HuggingFace + Transformers 让这件事的门槛低到“会 pip 就能上手”。

### HuggingFace Transformers 调用示例

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

# 1. 加载模型和分词器
model_id = "Qwen/Qwen1.5-0.5B-Chat"
device = "cuda" if torch.cuda.is_available() else "cpu"
tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForCausalLM.from_pretrained(
    model_id).to(device)

# 2. 准备对话
messages = [
    {"role": "system",
     "content": "You are a helpful assistant."},
    {"role": "user",
     "content": "你好，请介绍你自己。"}
]

# 3. 格式化并编码
text = tokenizer.apply_chat_template(
    messages, tokenize=False,
    add_generation_prompt=True)
inputs = tokenizer([text], return_tensors="pt").to(device)

# 4. 生成
out = model.generate(
    inputs.input_ids, max_new_tokens=512)

# 5. 解码
response = tokenizer.decode(
    out[0][inputs.input_ids.shape[1]:],
    skip_special_tokens=True)
print(response)
# 输出: 我叫通义千问，是由阿里云研发的预训练语言模型...
```

## 3.20 模型选型的关键考量

**大白话定义**：模型选型不是“谁最强就选谁”，而是综合你的任务、预算、隐私要求，挑出最合适的那一个。

**生活类比**：买车——跑长途选油车，市内通勤选电车，载货选皮卡。没有“最好”的车，只有“最适合”的车。

**七个常用维度**：
1. **能力**：写代码、推理、写作、多语言——各家侧重点不同。
2. **成本**：闭源按 token 付费，开源要自己买 GPU。
3. **速度**：客服、语音助手对延迟敏感，选轻量或优化版。
4. **上下文窗口**：长文档、长对话优先选大窗口模型。
5. **数据隐私**：医疗、金融、政企场景常要求本地部署。
6. **生态**：主流模型周边工具丰富（LangChain、LlamaIndex、Transformers），开发省力。
7. **可微调性**：需要垂直行业知识时，开源模型可以微调。

**和我有关系**：做 Agent 项目时，把这七条列成打分表，逐项打分再决策，比“听别人推荐”靠谱得多。

## 3.21 LLM 能力边界：会什么、不会什么

**大白话定义**：LLM 很强大，但它不是万能的。认清它的“擅长区”和“死角”，是把它用好的前提。

**它擅长什么**：
- 语言相关：写作、翻译、总结、改写、问答。
- 模式识别：从大量文本中学到的“套路”——生成代码、写邮件、做头脑风暴。
- 一步步推理：配合思维链提示，能做不少数学、逻辑题。

**它不擅长什么**：
- **幻觉（Hallucination）**：会一本正经地胡说八道——编造事实、引用假文献。
- **知识截止**：训练数据之后发生的事它不知道。
- **精确计算**：复杂数学、心算常常出错。
- **真正理解**：它本质是“接龙”，不是“思考”。
- **长尾知识**：偏门、小众领域知识可能完全错。

**和我有关系**：
- 关键决策、医疗、法律——必须人工核对，不能直接信 LLM。
- 用 RAG（检索增强）补知识、用工具调用补计算、用多步推理补逻辑——这是构建可靠 Agent 的三件套。

## 3.22 5 分钟体验：自己调一个提示词

**大白话定义**：光看理论不够，本节带你 5 分钟内“调教”一下 LLM，亲眼看到提示工程的效果。

**生活类比**：这就像学车的第一次上车——教练在副驾坐着，你亲自开一段。

**最小任务**（用任意 ChatGPT、文心、通义都能跑）：
**任务**：让模型把“客户反馈：服务慢，味道一般，但价格实惠”压缩成一句话总结。

**第 1 次：零样本**：
提示词：'请总结下面的客户反馈：服务慢，味道一般，但价格实惠。'
→ 模型可能写出 30 多个字的长句。

**第 2 次：加约束**：
提示词：'请把下面的客户反馈压缩成一句不超过 20 字的总结，突出"价格"是亮点：服务慢，味道一般，但价格实惠。'
→ 输出立刻短小精悍，重点突出。

**第 3 次：给格式**：
提示词：'请按 JSON 格式输出，包含"summary"和"highlight"两个字段。反馈：…'
→ 模型乖乖返回结构化数据，可以直接喂给程序。

**和我有关系**：一个提示词能影响输出质量、上线成本、开发难度。这就是为什么“会写提示词”成了 AI 时代最划算的技能之一。

## 3.23 本章小结

**一句话总结**：LLM 本质是“猜下一个字”的文字接龙高手，靠 Transformer 三件套（注意力 + 前馈 + 残差）堆叠而成，参数量越大能力越强。

**本章要点回顾**：
1. **LLM 是什么**：会读会写的文字预测器，本质是“猜下一个字”。
2. **为什么叫“大”**：参数量从 0.5B 到千亿级，量级直接决定能力上限。
3. **训练四步走**：预训练 → SFT → RLHF → 蒸馏，每一步解决不同问题。
4. **Transformer 三件套**：自注意力让每个字看到全文，前馈网络加工信息，残差+归一化让深层网络能训练。
5. **Decoder-Only 架构**：主流 LLM 都用这种“简洁到极致”的设计。
6. **提示工程**：角色、示例、思维链、约束——四个提升质量的常用技巧。
7. **采样参数与上下文窗口**：温度、Top-k、Top-p 控制输出风格，窗口大小决定它能记住多少内容。
8. **主流模型与多模态**：GPT/Claude/Gemini/文心/通义各有定位，多模态让 LLM 走出“只认文字”的限制。
9. **能力边界与可靠使用**：幻觉、知识截止、计算不准——靠 RAG、工具调用、多步验证来兜底。

下一章我们将把这一章学到的基础，应用到真正的智能体范式设计里——让 LLM 不只“会说话”，还能“会办事”。
