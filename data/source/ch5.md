第五章 基于低代码平台的智能体搭建

在前一章中，通过编写 Python 代码，我们从零开始实现了 ReAct、Plan-and-Solve 和 Reflection 多种经典的智能体工作流。这个过程为我们打下了坚实的技术基础，让我们深刻理解了智能体内部的运作机理。然而，对于一个快速发展的领域而言，纯代码的开发模式并非总是最高效的选择，尤其是在需要快速验证想法、或者非专业开发者希望参与构建的场景中。本章将聚焦三大国内最常见的低代码平台：Coze、Dify 和 FastGPT，并给出选型决策树和"低代码+代码"混合开发的实践范式。

5.1 什么是低代码平台

【大白话定义】低代码平台就是"用搭积木的方式做智能体"。你不用写 Python、不用懂 API，只要在网页上把一些现成的"模块"（比如知识库、对话节点、提示词、插件）拖一拖、连一连，就能拼出一个能听、能说、能查资料的 AI 应用。整个过程就像组装乐高——每个零件都封装好了，你只关心拼出什么造型。

【生活类比】把低代码平台想成"奶茶自助吧"：大模型是茶叶和奶，平台是搭配好的菜单和摇杯机，插件是珍珠、椰果、布丁这些小料。你只要选菜单按按钮，就能调出一杯属于自己的奶茶；不必像开奶茶店那样，先建工厂、研究配方。

【最小例子】最简流程只有 3 步：①新建一个"机器人"；②写一句角色设定，比如"你是一个客服，请礼貌回答售前问题"；③在插件市场里点一下"添加知识库"，上传一份 PDF 产品手册。点击发布，一个能根据手册回答问题的客服就上线了。

【和我有什么关系】如果你不是程序员、但想让 AI 帮自己干活——比如做客服、写小红书、查公司内部文档——低代码平台是门槛最低的入场券。它把第四章的 ReAct、Plan-and-Solve 都封装成可视化节点，你只要会"说话"就能指挥 AI。

5.1.1 为什么要用低代码（4 个理由）

【理由一：门槛低】产品经理、运营、客服主管都能上手，复杂 API 调用被封装成"拖拽"动作，再也不用求人写代码。

【理由二：出活快】从想法到可演示的原型，往往只要几小时；过去写代码可能要一两周。Coze 的口号就是"5 分钟搭一个 Bot"，并非夸张。

【理由三：看得见的调试】平台把智能体内部的每一步都画成流程图：哪个节点慢、哪个工具报错、哪段提示词没生效，一目了然。第四章里 print 大段日志的时代过去了。

【理由四：沉淀最佳实践】平台内置了行业模板（客服、写作、翻译、代码助手），相当于白送一套经过验证的工作流；新人来了直接套模板改，而不是从零摸索。

【和我有什么关系】这四个理由本质是把"开发智能体"从一项专业工程变成一项普及技能。即使你之后回到代码开发，低代码也能帮你快速验证想法，节省从 0 到 1 的时间。

5.1.2 三大平台横评：Coze / Dify / FastGPT

【大白话定义】这三位是国内最常被拿来比较的智能体低代码平台：Coze（扣子）来自字节跳动，主打"零代码 + 一键发布"；Dify 是开源的全栈平台，主打"专业、灵活、可私有化部署"；FastGPT 是国产开源项目，主打"知识库问答做得最深"。

【生活类比】把它们比作三种不同定位的餐厅：Coze 像必胜客，标准化程度高、点单就能吃、一键分享朋友圈，适合大众消费者；Dify 像开放厨房的西餐厅，菜单、食材、烹饪方式都能定制，适合有厨师基础的人；FastGPT 像专做老火汤的粤菜馆，把"炖、熬、提鲜"做到极致，专门解决"看文档回答问题"这件事。

【最小例子】同一道题"根据我们公司手册回答员工报销政策"，三家都能做：Coze 只需添加一个知识库+写一句人设；Dify 要新建 Chatflow + RAG Pipeline + 选择嵌入模型；FastGPT 走"知识库→应用"两步，专注做文档解析和检索排序。

【和我有什么关系】选哪个平台，决定你接下来 1-3 个月的工作效率和上线速度。错选轻则返工，重则项目烂尾。下面三节会带你一个个上手，再回到本章末给出决策树。

5.2 平台一：Coze

5.2.1 Coze 是什么：零代码智能体工厂

【大白话定义】Coze（扣子）是字节跳动在 2024 年推出的 AI Agent 平台，定位是"让完全不会写代码的人也能做出可发布的 AI 应用"。所有功能都做成可视化卡片：插件、知识库、工作流、卡片、提示词，像玩模拟经营游戏一样组建你的机器人。

【生活类比】把 Coze 想象成"AI 界的乐高乐园"。它不仅提供乐高积木（智能体模块），还提供大量"主题套装"（官方模板）、"专属零件"（插件商店）、"游乐设施"（一键发布到飞书/抖音/微信）。

【最小例子】搭一个"周报小助手"只需要：①新建 Bot 写人设"你是一个帮员工润色周报的助理"；②添加一个时间插件获取当前日期；③写提示词模板把员工输入的草稿润色成结构化周报；④点击发布到飞书。整套流程从注册到上线 10 分钟以内。

【和我有什么关系】如果你的目标是"快速做出可分享的 AI 工具并让真人用起来"，Coze 是目前最短的路径。它的一键发布渠道在国内生态里有不可替代的优势，做 ToC 产品的同学尤其受益。

5.2.2 Coze 实战：5 分钟搭一个客服

【最小例子】场景：一家做 SaaS 工具的小公司，客服每天被反复问"怎么开通试用、怎么开发票"。我们用 Coze 在 5 分钟内搭一个自动客服 Bot，步骤如下。

【第 1 步：建 Bot（1 分钟）】进入 coze.cn，点击"创建 Bot"，名称写"产品咨询助手"，头像选个机器人图标，进入编辑页。

【第 2 步：写人设和提示词（1 分钟）】在"人设与回复逻辑"里写："你是一名耐心的 SaaS 产品客服，只回答与产品试用、计费、发票相关的问题；其他问题礼貌引导至人工。"在左侧开场白里写"您好，我是产品咨询助手，请问有什么可以帮您？"

【第 3 步：加知识库（2 分钟）】点击"知识库"→"创建知识库"→上传一份 PDF《产品常见问题 30 问》；Coze 会自动切片、向量化、索引，1 分钟内完成。

【第 4 步：测试并发布（1 分钟）】在右侧对话框输入"怎么开发票？"，确认 Bot 引用了知识库片段并给出准确回答；点击"发布"，选择"飞书/抖音/微信小程序"等渠道。

【和我有什么关系】这个 5 分钟范式可以套用到几乎所有"基于文档回答问题"的场景：HR 政策、客服 FAQ、产品手册、培训资料。掌握之后，你不需要任何外包就能让 AI 替岗 1 个初级客服。

5.2.3 Coze 的优势与局限

【优势一：发布渠道广】一键发布到飞书、抖音、微信、豆包、网页插件等国内主流平台，是目前发布链路最完整的平台，做 C 端产品几乎绕不开。

【优势二：插件和模板丰富】官方插件库覆盖搜索、生图、语音、文件解析等上百个能力；模板市场里有大量可一键复制的成熟 Bot，"站在巨人肩膀上"做创新。

【优势三：上手最快】零代码的拖拽体验、清晰的引导教程、新手训练营，让完全没接触过 AI 的人也能在半小时内做出第一个可用的 Bot。

【局限一：MCP 支持滞后】Coze 自有插件体系完善，但目前对 MCP（Model Context Protocol）支持有限，要接入大量外部工具时需要走平台审核。

【局限二：复杂逻辑需绕弯】超过 20 个节点的工作流会出现维护困难；要写复杂判断、循环、外部 API 串接时，往往不如 Dify 直观。

【局限三：导出能力弱】付费版支持导出工作流 ZIP，但不是行业标准的 JSON 格式，迁移到自建系统需要重写。

【和我有什么关系】当你需要快速验证"AI 能否在这个业务上跑通"，Coze 是最高效的试验田；但当业务跑通、要进入生产时，要考虑是否需要迁到更专业的平台。

5.3 平台二：Dify

5.3.1 Dify 是什么：开源的全栈 AI 平台

【大白话定义】Dify 是一个开源的 LLM 应用开发平台，把"模型管理 + 提示词编排 + RAG 检索 + Agent 工作流 + 监控运营"打包成一个一站式平台。它的核心理念是 LLMOps——把 AI 应用当作软件工程一样规范化管理。

【生活类比】Dify 像一套"AI 时代的中台基础设施"。如果说 Coze 是装修好的样板间可以直接入住，Dify 则是开放工地的脚手架+建材库：你可以按需搭出 Chatflow、Workflow、Agent 多种房型，并接入自选品牌的"家电"（模型）。

【最小例子】在 Dify 中搭一个"小红书爆款标题生成器"：①新建 Chatflow；②添加"输入变量 title_keywords"和 LLM 节点；③写系统提示词"你是小红书爆款标题专家"+ Few-shot 示例；④连一个结束节点返回结果；⑤点击发布为 API。整套 5 步完成，得到的不是 Bot 而是一个 HTTP 接口，可以嵌入到任何业务系统。

【和我有什么关系】Dify 适合"需要把 AI 嵌入到已有业务系统、并且对数据安全有要求"的团队。它支持本地部署（Docker Compose 一键起），企业可以把数据留在自己机房；这一点是 Coze 给不了的。

5.3.2 Dify 实战：搭一个 RAG 问答

【最小例子】场景：一家律所希望做一个"合同风险点检索助手"，律师上传合同后能秒答"这份合同有哪些违约风险"。我们用 Dify 搭一个 RAG 应用。

【第 1 步：知识库（5 分钟）】进入"知识库"→"创建知识库"→上传 5 份历史合同 PDF；Dify 自动解析、分块、向量化，并支持多种分段模式（通用/父子/QA）。

【第 2 步：RAG Pipeline（5 分钟）】在工作室里新建"Chatflow"，拖入：开始节点 → 知识检索节点（绑定上一步的知识库） → 上下文拼接节点（把检索片段+用户问题合并）→ LLM 节点（系统提示词写"你是一名资深律师，基于以下合同片段回答风险点，并标注来源"）→ 直接回复节点。

【第 3 步：试运行（3 分钟）】在右侧对话框输入"这份合同有没有排他性条款？"，可以看到 LLM 不仅给出答案，还把引用到的合同片段和来源文件名展示出来。

【第 4 步：嵌入业务（5 分钟）】点击"发布"→"API 访问"，复制 cURL 命令到律所内部系统的前端按钮背后，律师点一下就能调起。

【和我有什么关系】RAG 是企业落地 LLM 最高频的场景，Dify 把这件事拆成"上传文档→建流程→发布 API"三步，比 LangChain 写 200 行代码快得多，而且可视化更便于和业务方对齐。

5.3.3 Dify 的优势与局限

【优势一：模型中立】Dify 不绑定任何模型供应商，OpenAI、Anthropic、DeepSeek、通义、文心、Llama 都行，企业可随时切换或做 AB Test，避免被一家供应商锁定。

【优势二：私有化部署】官方提供 Docker Compose 镜像，5 条命令就能在企业内网跑起来；金融、法律、医疗对数据出域敏感，Dify 是首选。

【优势三：插件市场庞大】Marketplace 上有 8000+ 插件和工具策略，覆盖搜索、数据库、向量库、MCP 工具，是目前生态最丰富的开源 LLM 平台之一。

【局限一：学习曲线较陡】节点众多、概念偏技术，新手面对 Chatflow 和 Workflow 两种模式容易懵，需要 1-2 天熟悉。

【局限二：高并发性能有瓶颈】核心后端是 Python 实现，吞吐相比 Go/Rust 方案有差距；大流量场景需要做集群和缓存优化。

【局限三：API 格式非 OpenAI 标准】第三方工具若要对接，需要写一层适配器，迁移成本比想象中多一点。

【和我有什么关系】当你开始把 AI 从 demo 推向生产，Dify 几乎是最稳的台阶——既能保持开发速度，又有企业级安全和扩展性。

5.4 平台三：FastGPT

5.4.1 FastGPT 是什么：把知识库问答做到极致

【大白话定义】FastGPT 是一个国产开源的 RAG 引擎和智能体平台，专注于"让知识库问答又快又准"。相比 Coze/Dify 的多功能集成，FastGPT 把文档解析、向量化、混合检索、重排序、问答评估这几件事打磨得更深。

【生活类比】如果说 Coze 是大众化的奶茶店、Dify 是开放式厨房的西餐厅，FastGPT 更像一个"只做手冲咖啡"的精品店——品类单一，但出品质量极高。它不追求插件数量，而追求"你给一份脏乱差的 PDF，它也能精准地回答问题"。

【最小例子】搭一个"公司制度问答助手"：①在 FastGPT 中新建知识库，上传《员工手册》《报销制度》《休假政策》3 份 Word；②选择"QA 拆分模式"自动抽取问答对；③新建"应用"绑定知识库，选择 DeepSeek 作为对话模型，调节 top-k、score 阈值等参数；④发布为 Web 应用或 API。实测在文档表述模糊、口语化提问时，FastGPT 命中率明显高于通用方案。

【和我有什么关系】如果你做的就是"看文档回答问题"——客服、培训、企业内部知识库——FastGPT 是当前国内最专业的选择，文档处理和检索排序的细节是它真正的护城河。

5.4.2 FastGPT 实战：知识库接入

【最小例子】场景：一家教育公司有 2000 份课程讲义 PDF，想做一个"AI 课程顾问"回答家长"这门课讲什么、适不适合我家孩子"。我们用 FastGPT 一步步接好。

【第 1 步：建知识库（10 分钟）】点击"知识库"→"新建"，上传讲义 PDF；FastGPT 会智能识别文档结构（标题、目录、章节），按"父子分段"切块（父块保留上下文，子块用于精确检索），可以预览每个 chunk 看到文本质量。

【第 2 步：配检索策略（5 分钟）】进入"应用"→新建"知识库+对话"应用；选择"混合检索"= 向量检索 + 全文检索，再开"重排序"用一个 Cross-Encoder 模型把召回的 top-20 重排成 top-3，准确率通常提升 20%+。

【第 3 步：接 API 到业务（5 分钟）】在"应用"中点击"发布"→"API 接入"，复制接口地址和鉴权 key；用 5 行 Python 代码把请求接到公司公众号后端，公众号收到家长提问时转发给 FastGPT，把答案原文返回。

【第 4 步：调优（持续）】FastGPT 自带"评测"功能，可以批量跑测试集看命中率、Citation 准确率，并对比不同 embedding 模型的效果。

【和我有什么关系】这套流程可以套到所有"文档密集型"业务：法律合同、产品手册、医疗指南、企业 SOP。FastGPT 的优势在于：同样的文档，它问得最准；省下来的人力成本往往远超平台学习成本。

5.4.3 FastGPT 的优势与局限

【优势一：RAG 最专业】文档解析、混合检索、QA 拆分、重排序、评测一应俱全；同类问题命中率明显高于通用平台。

【优势二：国产开源 + 社区活跃】中文文档友好、GitHub Star 1.5w+、微信群/Discord 答疑响应快，遇到问题基本 24 小时内有答案。

【优势三：可商用、可私有化】Apache 2.0 协议，Docker 一键部署，企业可放心用于商业项目并部署在内网。

【局限一：Agent 能力相对弱】相比 Coze/Dify，FastGPT 的"工具调用、多智能体、复杂工作流"能力较弱，做任务型 Agent 不如前者灵活。

【局限二：发布渠道少】不像 Coze 一键接飞书/抖音；FastGPT 主要是 Web 应用和 API，分发需要自己开发。

【局限三：UI 偏工程师审美】界面信息密度大、配置项多，对纯非技术用户不太友好，更适合有开发者协作的团队。

【和我有什么关系】如果你的项目核心是"文档问答"，首选 FastGPT；如果核心是"任务型 Agent 或多渠道分发"，回到 Coze/Dify。下一节会用一张对比表把这些差异量化。

5.5 三家对比决策树

5.5.1 决策树：个人 / 团队 / 企业怎么选

【决策第一步：你是谁？】三种典型用户画像，对应完全不同的最优解。

【个人创作者 / 运营 / 产品经理】痛点是"想快速出东西、不想碰代码"。优先选 Coze：5 分钟出 Bot，一键发布到抖音/小红书/微信，模板丰富。

【10-50 人创业团队 / 独立开发者】痛点是"业务跑通但要可控、要能接 API"。优先选 Dify：私有化部署+模型中立+插件多，能从 demo 顺滑过渡到生产。

【企业 IT 部门 / 客服中心 / 知识密集型团队】痛点是"文档多、要精准、不能上云"。优先选 FastGPT：把 RAG 做到极致，本地化部署简单。

【决策第二步：业务是什么类型？】
• 创意分发、对话 Bot、营销内容 → Coze
• 复杂工作流、Agent 任务、模型可换 → Dify
• 知识库问答、客服、文档检索 → FastGPT

【决策第三步：兜底原则】如果还是选不出来，记住一个心法：先 Coze 验证想法，再迁到 Dify/自建做生产。FastGPT 单独评估 RAG 是否是关键瓶颈。

【和我有关系】这一节是本章最值钱的"选择模板"。下次要做新项目时拿出来对照，能省 2-4 周调研时间。

5.6 低代码 vs 自建：什么时候该上代码

【大白话定义】"上代码"指的是回到第四章那种用 Python 直接调模型 API 的方式。"低代码"则是用 Coze/Dify/FastGPT 拖拽搭建。两者不是对立的，而是同一项目的不同阶段。

【生活类比】把开发智能体比作做菜。低代码是"半成品菜+智能炒菜机"：标准化场景 3 分钟出锅；自建是"从挑菜、洗菜、切菜、调味到摆盘"，全程自己控制味道、品质和创新空间。

【该用低代码的 3 个场景】
①需求验证期：用 1-3 天搭 demo 给业务方看，跑通就上，跑不通就换；
②标准化场景：客服 FAQ、文案润色、文档问答，模板已经成熟；
③人手紧且无算法工程师：产品/运营主导，让 AI 帮他们干活。

【该自建的 3 个场景】
①核心业务壁垒：低代码平台的提示词和工作流逻辑都在平台方手里，做核心产品要可控；
②极致性能/成本：百万级 QPS 或成本敏感场景，需要定制模型路由、缓存、并发；
③深度行业集成：要调内部 ERP/CRM/SCM 系统的私有 API，平台插件支持有限。

【和我有什么关系】正确的姿势是"低代码 + 代码"的混合架构：80% 的标准化流程交给平台，20% 的差异化逻辑用代码补；既快又稳。

5.7 跟着做：3 小时搭一个完整客服

【目标】从零开始，3 小时内搭出一个能在公众号里回答公司产品问题的智能客服，体验"低代码 + 代码"的混合流程。

【第 1 小时：用 Coze 搭 Bot 骨架】注册 Coze → 创建"产品咨询助手" → 写人设 + 上传 3 份产品 PDF 知识库 → 添加"转人工"卡片 → 一键发布到飞书机器人，验证对话流程。

【第 2 小时：用 Dify 搭后端 RAG】本地用 Docker Compose 起一套 Dify → 同步上传同样的产品文档 → 搭 Chatflow：知识检索 → LLM → 回复 → 打开 API 文档，复制 endpoint 和 key。

【第 3 小时：用 Python 串起来】写 30 行 Flask 小服务：监听飞书 Webhook → 调 Dify API → 把答案原文回给用户。中间穿插"意图识别"代码段：检测到关键词"投诉/退款"时，强制路由到人工队列。

【产出物】一个能在飞书/网页中使用的智能客服，文档更新两边同时生效（Coze 知识库 + Dify 知识库），关键流程有 Python 兜底。

【和我有关系】这是把本章所有概念串起来的小项目，做完后你就掌握了"低代码搭主体 + 代码补差异"的标准范式，未来可平移到任何业务场景。

5.7.1 提示词与工作流设计原则

【原则一：人设要具体】"你是一个专业、耐心、幽默的客服"比"你是一个好的客服"有效得多。要写明身份、能力边界、回答风格、不能做什么。

【原则二：变量用 {{}} 占位】Coze/Dify/FastGPT 都支持提示词模板，用 `{{user_question}}` 这种变量占位，可以把用户输入、知识库片段、工具返回值动态拼到提示词里。

【原则三：Few-shot 优于描述】与其写 200 字教模型"请这样回答"，不如直接给 2-3 个问答示例，模型照葫芦画瓢效果更稳。

【原则四：拆工作流 > 一锅端】复杂任务要拆成"意图识别 → 分支 → 工具调用 → 汇总"多个节点；一锅端会让 LLM 容易跑偏，单步可观测、可重试。

【原则五：加日志和兜底】每个工作流节点都要把输入输出打到日志；关键步骤加"if 失败则重试或回退"逻辑，避免一个工具挂掉整个流程崩。

【原则六：评测驱动迭代】准备 20-50 条测试用例（覆盖正常、异常、边界场景），每次改完提示词或工作流先跑一遍评测，看指标变化再上线。

【和我有关系】这 6 条是从大量失败项目中沉淀出来的"避坑清单"，第一次做智能体的人几乎一定会踩其中几条；提前看可以少走 1-2 周弯路。

5.7.2 上线运营与避坑指南

【坑一：知识库过期】产品迭代后老文档没更新，Bot 答的是过期信息。解法：建立"文档版本→知识库同步"的发布流水线，每次发版自动重建索引。

【坑二：幻觉率失控】Bot 在没有知识库覆盖的问题上自由发挥。解法：给提示词加"如果不知道就回答'我没有该信息，请联系人工'"，并设置"答案置信度阈值"。

【坑三：token 成本爆表】长文档全量塞上下文，单次对话烧几毛钱。解法：把检索 top-k 控制在 3-5，分块大小设 500-800 token，开启上下文压缩。

【坑四：用户问敏感问题】Bot 在金融、医疗、法律领域可能给出不当建议。解法：领域限定提示词 + 关键词拦截 + 高危场景转人工。

【坑五：平台锁定】业务做大后想迁移到自建，发现工作流导不出来。解法：核心工作流定期用 JSON 备份；关键提示词放在自己的代码仓库管理。

【坑六：评估缺失】上线后不知道效果好不好。解法：搭建用户反馈按钮（👍/👎）+ 周报看命中率/转人工率/平均对话轮次。

【和我有关系】这些是真实项目里反复出现的问题，预先知道能少踩 80% 的坑；尤其"幻觉率"和"成本"是老板必问的指标，必须有应对方案。

5.8 本章小结

【一句话总结】低代码平台把第四章的代码活变成了"搭积木"，让产品、运营、客服等非技术同学也能做出生产可用的 AI 应用；Coze、Dify、FastGPT 三家各有侧重，是当前国内最值得掌握的智能体开发平台。

【核心收获】
①低代码的核心价值是"降低门槛 + 提升效率 + 沉淀最佳实践"，不是取代代码；
②Coze 强在发布渠道和零代码体验，适合 C 端创意类应用；
③Dify 强在模型中立和私有化部署，适合企业级生产系统；
④FastGPT 强在知识库问答的极致优化，适合文档密集型业务；
⑤选型没有标准答案，先用 Coze 验证，再用 Dify/自建做生产；
⑥"低代码 + 代码"混合架构是工程化的最佳实践。

【下一步】第六章我们将学习 LangChain、AutoGen 等智能体开发框架，把这一章的"拖拽实现"升级为"代码级掌控"，应对更复杂、更高性能要求的场景。

【和我有关系】这一章是承上启下的关键：上承第四章的代码原理，下启第六章的框架开发。掌握本章的"低代码思维"后，再写代码会觉得所有能力都被平台化、模块化封装好了。
